# Генетический Алгоритм Маршрутизации: Подробное Объяснение

## 1. Что именно оптимизирует алгоритм

Система решает не задачу построения дороги с нуля, а задачу **поиска лучшего порядка обхода заданных пользователем точек**.

Если пользователь задал точки:

```text
P0, P1, P2, ..., P(n-1)
```

то алгоритм ищет такую перестановку этих точек, при которой итоговый маршрут лучше по нескольким критериям:

- расстояние,
- длительность,
- стоимость топлива,
- выбросы CO2,
- загруженность,
- погодный риск,
- надёжность,
- безопасность,
- платные дороги.

Модель запроса и параметры запуска описаны в [app/domain/models.py](../app/domain/models.py).

## 2. Как кодируется маршрут в генетическом алгоритме

### 2.1. Случай с фиксированным стартом и финишем

Если включено `fix_ends = true`, то:

- первая точка всегда остаётся стартом,
- последняя точка всегда остаётся финишем,
- переставляются только внутренние точки.

Тогда хромосома имеет вид:

```text
[0] + permutation(1..n-2) + [n-1]
```

Пример для 8 точек:

```text
0, 1, 2, 3, 4, 5, 6, 7
```

Если геном:

```text
(3, 1, 5, 2, 6, 4)
```

то полный маршрут после декодирования:

```text
0 -> 3 -> 1 -> 5 -> 2 -> 6 -> 4 -> 7
```

Именно так работает `_decode(...)` в [app/services/route_optimizer.py](../app/services/route_optimizer.py).

### 2.2. Случай без фиксированных концов

Если `fix_ends = false`, то геномом становится перестановка всех точек:

```text
permutation(0..n-1)
```

### 2.3. Что это значит для вашего текущего сценария

Если точек `8`, а старт и финиш фиксированы, переставляются `6` внутренних точек.

Число возможных порядков:

```text
6! = 720
```

То есть даже для 8 точек задача уже не тривиальна, и генетический алгоритм здесь уместен.

## 3. Какие данные строятся перед запуском GA

Перед оптимизацией система собирает `OptimizationContext` в [app/services/context_service.py](../app/services/context_service.py).

В этот контекст входят:

- `distance_matrix_km`
- `duration_matrix_min`
- `traffic_matrix`
- `toll_matrix`
- `weather_profiles`
- `elevation_gain_matrix_m`
- `elevation_loss_matrix_m`
- `mean_elevation_matrix_m`
- `departure_at`

### 3.1. Почему всё хранится в виде матриц

Пусть у нас есть `n` точек. Тогда почти любой критерий для маршрута можно оценивать попарно: для перехода из точки `i` в точку `j`.

Для этого удобно заранее построить квадратные матрицы размера:

```text
n x n
```

Каждая ячейка хранит стоимость перехода `i -> j`.

Например:

- `distance_matrix_km[i][j]` = расстояние в км от точки `i` до `j`
- `duration_matrix_min[i][j]` = базовое время в минутах
- `traffic_matrix[i][j]` = индекс загруженности
- `toll_matrix[i][j]` = стоимость платных дорог

Аналогично для рельефа:

- `elevation_gain_matrix_m[i][j]` = суммарный набор высоты на сегменте
- `elevation_loss_matrix_m[i][j]` = суммарный сброс высоты
- `mean_elevation_matrix_m[i][j]` = средняя высота по сегменту

### 3.2. Пример матриц на 4 точках

Пусть есть точки:

- `0 = S` — старт
- `1 = A`
- `2 = B`
- `3 = F` — финиш

#### Матрица расстояний

```text
           S     A     B     F
S [0]    0.0  10.0  18.0  30.0
A [1]   10.0   0.0  12.0  16.0
B [2]   18.0  12.0   0.0   9.0
F [3]   30.0  16.0   9.0   0.0
```

Например:

- `distance_matrix_km[0][1] = 10.0`
- `distance_matrix_km[2][3] = 9.0`

#### Матрица базового времени

```text
           S     A     B     F
S [0]    0.0  12.0  20.0  34.0
A [1]   12.0   0.0  16.0  18.0
B [2]   20.0  16.0   0.0  11.0
F [3]   34.0  18.0  11.0   0.0
```

#### Матрица трафика

```text
           S     A     B     F
S [0]   0.00  0.15  0.05  0.20
A [1]   0.15  0.00  0.10  0.25
B [2]   0.05  0.10  0.00  0.08
F [3]   0.20  0.25  0.08  0.00
```

#### Матрица платных дорог

```text
           S     A     B     F
S [0]    0.0   0.0  90.0 150.0
A [1]    0.0   0.0   0.0  40.0
B [2]   90.0   0.0   0.0   0.0
F [3]  150.0  40.0   0.0   0.0
```

#### Матрица набора высоты

```text
           S     A     B     F
S [0]      0    40    55    70
A [1]     40     0    35    20
B [2]     55    35     0    10
F [3]     70    20    10     0
```

#### Матрица сброса высоты

```text
           S     A     B     F
S [0]      0    15    30    45
A [1]     15     0    18    22
B [2]     30    18     0    25
F [3]     45    22    25     0
```

#### Матрица средней высоты

```text
           S     A     B     F
S [0]      0   150   165   140
A [1]    150     0   175   130
B [2]    165   175     0   120
F [3]    140   130   120     0
```

## 4. Как из матриц считается маршрут

Пусть пользовательский порядок точек:

```text
S -> B -> A -> F
```

Тогда массив индексов:

```text
[0, 2, 1, 3]
```

Маршрут состоит из трёх рёбер:

- `0 -> 2`
- `2 -> 1`
- `1 -> 3`

Для каждого ребра алгоритм просто читает нужные ячейки матриц:

```text
d1 = distance[0][2] = 18
d2 = distance[2][1] = 12
d3 = distance[1][3] = 16
```

И так же для времени, трафика, платных дорог, рельефа.

Суммарное расстояние маршрута:

```text
18 + 12 + 16 = 46 км
```

Если другой порядок:

```text
S -> A -> B -> F
```

то индексы:

```text
[0, 1, 2, 3]
```

Суммарное расстояние:

```text
10 + 12 + 9 = 31 км
```

Уже на одном только расстоянии видно, что второй маршрут лучше.

## 5. Как учитывается время и погода

Расчёт делается в `CriteriaService.evaluate(...)` в [app/services/criteria_service.py](../app/services/criteria_service.py).

### 5.1. Базовое время сегмента

Для сегмента `i -> j` берётся:

```text
tau_ij = duration_matrix_min[i][j]
```

### 5.2. Погода не одна на весь маршрут

Сейчас в системе погода учитывается реалистичнее:

- для каждой пользовательской точки заранее загружается hourly forecast,
- для каждого сегмента оценивается примерное время прохождения его середины,
- затем погода берётся на этот момент времени,
- погодный снимок сегмента усредняется по концам сегмента.

То есть для длинного маршрута условия могут меняться по мере движения.

### 5.3. Формула длительности сегмента

Если:

- `tau_ij` — базовое время,
- `c_ij` — загруженность сегмента,
- `w_ij` — погодная severity на момент прохождения сегмента,

то:

```text
weather_delay_factor = 0.35 * w_ij
duration_ij = tau_ij * (1 + c_ij + weather_delay_factor)
```

### 5.4. Наглядный пример

Для маршрута:

```text
S -> A -> B -> F
```

пусть погодная severity по сегментам:

- `S -> A`: `0.10`
- `A -> B`: `0.20`
- `B -> F`: `0.15`

Тогда:

#### Сегмент `S -> A`

```text
tau = 12
traffic = 0.15
weather = 0.10

duration = 12 * (1 + 0.15 + 0.35*0.10)
         = 12 * (1 + 0.15 + 0.035)
         = 12 * 1.185
         = 14.22 мин
```

#### Сегмент `A -> B`

```text
tau = 16
traffic = 0.10
weather = 0.20

duration = 16 * (1 + 0.10 + 0.35*0.20)
         = 16 * (1 + 0.10 + 0.07)
         = 16 * 1.17
         = 18.72 мин
```

#### Сегмент `B -> F`

```text
tau = 11
traffic = 0.08
weather = 0.15

duration = 11 * (1 + 0.08 + 0.35*0.15)
         = 11 * (1 + 0.08 + 0.0525)
         = 11 * 1.1325
         = 12.46 мин
```

Итог:

```text
total_duration = 14.22 + 18.72 + 12.46 = 45.40 мин
```

## 6. Как считаются риск, надёжность и безопасность

Для каждого сегмента считаются две отдельные характеристики:

- `reliability_risk`
- `safety_risk`

Формулы в коде:

```text
incident_proxy = clamp(0.1 + 0.25*congestion + 0.2*weather, 0, 1)

reliability_risk =
  clamp(0.45*congestion + 0.35*weather + 0.2*incident_proxy, 0, 1)

safety_risk =
  clamp(0.5*weather + 0.3*congestion + 0.2*night_factor, 0, 1)
```

### Пример

Для сегмента:

```text
congestion = 0.10
weather = 0.20
night_factor = 0.10
```

получаем:

```text
incident_proxy = 0.1 + 0.25*0.10 + 0.2*0.20
               = 0.1 + 0.025 + 0.04
               = 0.165
```

```text
reliability_risk = 0.45*0.10 + 0.35*0.20 + 0.2*0.165
                 = 0.045 + 0.07 + 0.033
                 = 0.148
```

```text
safety_risk = 0.5*0.20 + 0.3*0.10 + 0.2*0.10
            = 0.10 + 0.03 + 0.02
            = 0.15
```

По всему маршруту:

```text
reliability_score = 1 - average(reliability_risk)
safety_risk_avg   = average(safety_risk)
```

## 7. Как учитывается рельеф

Внутри оптимизатора рельеф не сводится к разнице высот только в концах сегмента.

Для пары точек `i -> j` заранее строятся:

- суммарный набор высоты,
- суммарный сброс высоты,
- средняя высота.

То есть если у двух опорных точек одинаковая высота, но между ними есть серия подъёмов и спусков, они всё равно попадут в расчёт.

### 7.1. Пример на маршруте `S -> A -> B -> F`

Из матриц:

- `gain(S,A) = 40`
- `gain(A,B) = 35`
- `gain(B,F) = 10`

Тогда:

```text
total_gain = 40 + 35 + 10 = 85 м
```

Аналогично:

- `loss(S,A) = 15`
- `loss(A,B) = 18`
- `loss(B,F) = 25`

```text
total_loss = 15 + 18 + 25 = 58 м
```

Если расстояние маршрута `31 км`, то:

```text
horizontal_m = 31 * 1000 = 31000 м

uphill_pct   = 85 / 31000 * 100 = 0.274%
downhill_pct = 58 / 31000 * 100 = 0.187%
```

Это не означает, что на каждом метре трассы уклон ровно такой.  
Это агрегированная доля накопленного набора и сброса относительно общей горизонтальной длины.

## 8. Как считается топливо

В оптимизаторе расход топлива зависит не только от расстояния.

Используются:

- базовый расход транспорта,
- рельеф,
- средняя высота,
- температура,
- загруженность.

Общая идея:

```text
liters =
  distance_km * consumption_l_per_100km / 100
  * terrain_multiplier
  * mountain_multiplier
  * temperature_multiplier
  * (1 + 0.2 * congestion_index)
```

Стоимость:

```text
fuel_cost = liters * price_per_liter + toll_cost
```

Выбросы:

```text
co2_kg = liters * emission_factor
```

## 9. Как работают ограничения

Ограничения задаются в `constraints`.

Если маршрут их нарушает, начисляется штраф:

```text
distance penalty = 1200 * relative_violation
duration penalty = 1200 * relative_violation
fuel penalty     = 1400 * relative_violation
co2 penalty      = 1100 * relative_violation
safety penalty   = 1600 * relative_violation
```

### Пример

Если задано:

```text
max_fuel_cost = 1000
```

а маршрут дал:

```text
fuel_cost = 1150
```

то:

```text
relative_violation = (1150 - 1000) / 1000 = 0.15
penalty = 1400 * 0.15 = 210
```

Такой маршрут становится заметно хуже в fitness-функции.

## 10. Как формируется objective score

### 10.1. Абсолютный score для одного маршрута

Если оценивается только один маршрут, используется абсолютная формула:

```text
score =
  w_distance * distance
  + w_duration * duration
  + w_fuel * fuel_cost
  + w_emissions * co2
  + w_congestion * congestion
  + w_weather * weather_risk
  + w_reliability * (1 - reliability_score)
  + w_safety * safety_risk
  + w_tolls * toll_cost
  + constraint_penalty
```

### 10.2. Score внутри популяции GA

Если маршрутов много, то метрики сначала нормализуются внутри текущего набора кандидатов:

```text
norm(x) = (x - min_population) / (max_population - min_population)
```

А затем:

```text
score =
  w_distance * norm(distance)
  + w_duration * norm(duration)
  + w_fuel * norm(fuel_cost)
  + w_emissions * norm(co2)
  + w_congestion * norm(congestion)
  + w_weather * norm(weather_risk)
  + w_reliability * norm(1 - reliability_score)
  + w_safety * norm(safety_risk)
  + w_tolls * norm(toll_cost)
  + norm(constraint_penalty)
```

Важно:

- такой `objective_score` удобен для эволюционного отбора,
- но его нельзя корректно сравнивать между двумя разными запусками,
- для анализа лучше смотреть raw-метрики маршрута.

## 11. Что означают веса критериев

Веса задаются пользователем в форме и хранятся в `CriteriaWeights`.

По умолчанию в проекте:

```text
distance     = 1.0
duration     = 1.2
fuel_cost    = 1.1
emissions    = 0.9
congestion   = 1.0
weather_risk = 0.8
reliability  = 0.9
safety       = 1.0
tolls        = 0.4
```

Сумма:

```text
8.3
```

После нормализации:

| Критерий | Вес | Нормализованный вклад |
|---|---:|---:|
| Distance | 1.0 | 12.05% |
| Duration | 1.2 | 14.46% |
| Fuel cost | 1.1 | 13.25% |
| Emissions | 0.9 | 10.84% |
| Congestion | 1.0 | 12.05% |
| Weather risk | 0.8 | 9.64% |
| Reliability | 0.9 | 10.84% |
| Safety | 1.0 | 12.05% |
| Tolls | 0.4 | 4.82% |

### Интерпретация

Для текущих настроек:

- важнее всего `duration`,
- затем `fuel_cost`,
- затем группа `distance / congestion / safety`,
- наименее важны `tolls`.

## 12. Как dynamic weights меняют пользовательские веса

Система может не использовать веса из формы буквально.  
Сначала она применяет профиль приоритета:

- `fastest`: усиливает `duration` и `congestion`
- `cheapest`: усиливает `fuel_cost` и `tolls`
- `safest`: усиливает `safety`, `reliability`, `weather_risk`
- `greenest`: усиливает `emissions` и немного `fuel_cost`

После этого могут сработать контекстные триггеры:

- `peak_hour`
- `bad_weather`
- `high_congestion`
- `high_fuel_price`

И только потом веса снова нормализуются.

То есть если в форме стоят одинаковые числа, это ещё не означает, что именно с ними пойдёт реальный расчёт.

## 13. Как формируется начальная популяция

В популяцию попадают:

- исходный порядок пользователя,
- маршрут из жадной эвристики `nearest neighbor`,
- случайные перестановки.

### Наглядный пример

Для точек:

```text
S, A, B, F
```

и `fix_ends = true` возможны только два внутренних порядка:

- `(A, B)` -> маршрут `S -> A -> B -> F`
- `(B, A)` -> маршрут `S -> B -> A -> F`

Если точек больше, пространство резко растёт, и случайные перестановки начинают играть большую роль.

## 14. Как работают selection, crossover и mutation

### 14.1. Отбор родителей

В weighted-режиме:

- сравниваются две случайные особи,
- сначала предпочитается допустимая `feasible`,
- затем та, у которой меньше `objective_score`.

В pareto-режиме:

- сначала лучше меньший `rank`,
- затем больше `crowding`,
- затем меньше `objective_score`.

### 14.2. Кроссовер

Используется `Order Crossover`.

Пример:

```text
parent A = (1, 3, 5, 2, 6, 4)
parent B = (3, 2, 4, 6, 1, 5)
```

Пусть выбран фрагмент `[2..4]` из первого родителя:

```text
child = (_, _, 5, 2, 6, _)
```

Оставшиеся позиции заполняются генами второго родителя по порядку, пропуская уже использованные:

```text
child = (4, 1, 5, 2, 6, 3)
```

Смысл операции:

- сохраняется корректная перестановка,
- сохраняются локальные блоки хороших решений,
- не возникают дубликаты точек.

### 14.3. Мутация

В коде есть два варианта:

- swap двух точек,
- reverse подотрезка.

Пример swap:

```text
(1, 3, 5, 2, 6, 4) -> (1, 6, 5, 2, 3, 4)
```

Пример reverse:

```text
(1, 3, 5, 2, 6, 4) -> (1, 2, 5, 3, 6, 4)
```

## 15. Что значат ваши параметры GA

Если в форме стоят:

```text
population_size = 96
generations     = 120
crossover_rate  = 0.88
mutation_rate   = 0.22
max_alternatives = 8
```

то это означает:

- `population_size = 96`  
  В каждом поколении храним 96 кандидатов.

- `generations = 120`  
  Делаем 120 итераций эволюции.

- `crossover_rate = 0.88`  
  В 88% случаев потомки будут собраны из двух родителей, а не просто скопированы.

- `mutation_rate = 0.22`  
  Около 22% потомков дополнительно случайно изменятся.

- `max_alternatives = 8`  
  Пользователю показываем до 8 Pareto-альтернатив.

### Практическая интерпретация

Для маршрута на 8 точках эти параметры выглядят разумно:

- популяция уже достаточно разнообразна,
- поколений достаточно для стабилизации,
- мутация не слишком слабая,
- кроссовер используется активно.

## 16. Чем отличается weighted от pareto

### Weighted

Оптимизатор сводит всё к одному числу:

```text
objective_score
```

Это удобно, когда нужно выбрать один лучший маршрут.

### Pareto

Оптимизатор не пытается сразу всё сложить в один показатель, а ищет набор недоминируемых решений.

Решение `A` доминирует `B`, если:

- по всем критериям `A` не хуже `B`,
- и хотя бы по одному критерию `A` строго лучше.

Это даёт фронт компромиссов:

- один маршрут может быть быстрее,
- другой дешевле,
- третий безопаснее.

## 17. Что пользователь видит в результате

После оптимизации сервис считает:

- `optimized` — лучший найденный маршрут,
- `baseline` — маршрут в исходном порядке пользователя.

Затем строится сравнение:

- разница в км,
- разница во времени,
- разница в стоимости топлива,
- разница по CO2,
- разница по `objective_score`.

Важно:

- блок сравнения всегда идёт с `baseline`,
- а не с той альтернативой, которая выбрана на карте в данный момент.

## 18. Короткий ручной пример целиком

Пусть есть два кандидата:

- базовый: `S -> B -> A -> F`
- оптимизированный: `S -> A -> B -> F`

Из матриц получаем:

### Базовый

```text
distance = 18 + 12 + 16 = 46 км
```

### Оптимизированный

```text
distance = 10 + 12 + 9 = 31 км
```

Улучшение по расстоянию:

```text
(46 - 31) / 46 * 100 = 32.61%
```

Если и время, и стоимость топлива, и риски у второго варианта тоже ниже, именно он получит лучший fitness и выживет в популяции.

## 19. Что особенно важно объяснить научному руководителю

1. Генетический алгоритм работает не с геометрией дороги напрямую, а с перестановками точек маршрута.
2. Для быстрой оценки используются матрицы попарных переходов между точками.
3. Оценка маршрута многокритериальная: время, стоимость, риски, экология, рельеф.
4. Погода уже учитывается не только в момент старта, а по примерному времени прохождения сегментов.
5. Ограничения включены через штрафную функцию.
6. В weighted-режиме fitness внутри популяции относительный, поэтому для научного анализа важнее raw-метрики маршрутов и Pareto-альтернативы.
